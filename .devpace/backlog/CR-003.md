## 意图

- **用户原话**：推进 PF-003，ALB + HTTPS + WAF 替代直接端口暴露 <!-- source: user -->
- **范围**：<!-- source: claude -->
  - 做什么：Application Load Balancer 前置代理 OpenClaw 三个端口；ACM 证书参数化支持 HTTPS（无证书时 HTTP fallback）；WAF WebACL 关联 ALB（AWS Managed Core Rule Set）；EC2 SG 收紧为仅接受 ALB 流量；CfnOutput 输出 ALB endpoint
  - 不做什么：不注册域名/Route53（用户自行管理 DNS）；不配置 CloudFront CDN；不实现自定义 WAF 规则（首迭代用托管规则集）
- **验收条件**：<!-- source: claude -->
  1. Given ALB 已创建, When 检查合成模板, Then 包含 ALB + Target Group + HTTP Listener 资源
  2. Given ACM 证书 ARN 已提供, When 检查模板, Then 包含 HTTPS Listener（443）+ HTTP→HTTPS 重定向
  3. Given WAF WebACL 已创建, When 检查模板, Then WebACL 关联到 ALB，含 AWSManagedRulesCommonRuleSet
  4. Given SG 策略已优化, When 检查 EC2 SG, Then OpenClaw 端口仅接受 ALB SG 流量（非公网 CIDR）
  5. Given cdk synth 通过, When 运行合成, Then 0 错误
- **方案**：<!-- source: claude -->
  - ALB 位于公有子网，EC2 保持在默认 VPC
  - HTTPS 可选：提供 `certificateArn` 参数时启用 443 + 重定向；未提供时仅 HTTP
  - WAF 使用 AWS Managed Rules（AWSManagedRulesCommonRuleSet + AWSManagedRulesKnownBadInputsRuleSet）
  - EC2 SG 改为仅允许 ALB SG + SSH（管理端口保留直连）
- **约束**：
  - HTTPS 需要有效的 ACM 证书 ARN（用户在目标区域预先创建）
  - WAF 在 us-east-1 以外区域可能有功能差异
  - ALB 需要至少 2 个可用区的子网

### 执行计划

1. **`lib/constructs/alb.ts`**：ALB + Target Group + Listeners
   - 预期：ALB 公有子网，Target Group 转发到 EC2 三个端口，HTTP Listener（80），可选 HTTPS Listener（443）
   - 命令：`npx tsc --noEmit`

2. **`lib/constructs/waf.ts`**：WAF WebACL + AWS Managed Rules
   - 预期：WebACL 含 2 个 AWS 托管规则组，关联到 ALB ARN
   - 命令：`npx tsc --noEmit`

3. **更新 `networking.ts`**：SG 策略优化
   - 预期：EC2 SG OpenClaw 端口改为仅接受 ALB SG，SSH 保留 allowedCidr 直连
   - 命令：`npx tsc --noEmit`

4. **更新 `clawforce-stack.ts`**：集成 ALB + WAF + 更新 Outputs
   - 预期：Stack 新增 ALB/WAF 构建体，CfnOutput 包含 ALB DNS + URL
   - 命令：`npx cdk synth`

5. **验证**：cdk synth + 资源完整性检查
   - 预期：CloudFormation 模板含 ALB、WAF、更新的 SG 资源
   - 命令：`npx cdk synth | grep -c 'AWS::ElasticLoadBalancingV2\|AWS::WAFv2'`
